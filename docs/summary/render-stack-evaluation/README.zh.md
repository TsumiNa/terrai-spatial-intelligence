# 渲染栈评估：CesiumJS 与 MapLibre + deck.gl

[English](README.md) | [日本語](README.ja.md) | [中文](README.zh.md)

- 日期：2026-07-20
- 状态：已决定——地图采用 MapLibre GL JS + deck.gl，场址场景采用独立 Three.js
- 范围：面向计划中的前端重构，对浏览器端渲染栈的评估

## 为什么做这次评估

Leaflet 只渲染 2D 栅格瓦片。路线图要加 3D 地形和地下可视化，而这两件事它根本无法表达。针对同一地区、同一份 `building_risk.geojson`（2,128 栋建筑）、同一套 GSI 数据源，构建了两个用完即弃的 spike。

本文记录实测结果，以及这些结果导向的决定。

## GSI 数据源的层级上限

针对两个演示矩形直接探测所得。

| 图层 | 最深可用层级 | 备注 |
|---|---|---|
| `slopemap` | z15 | z16 返回 404 |
| `hillshademap` | z16 | z17 返回 404 |
| `seamlessphoto` | z18 | z19 返回 404 |
| `ort` | z18 | 同层级下约 57 KB/瓦片，seamlessphoto 约 17 KB |
| `dem5a_png` | z15 | 数值高程，两个演示区均可用 |
| `dem_png` | z14 | DEM10B 回退 |
| `experimental_bvmap` | z16 | 矢量瓦片，官方提供 774 图层的样式 |

GSI 瓦片一律返回 `access-control-allow-origin: *`，因此可以在浏览器中解码其像素。

## 地面分辨率，以及栅格坡度图为何无法变清晰

北纬 35.45° 处：

| 层级 | 分辨率 |
|---|---|
| z15 | 3.89 m/px |
| z16 | 1.95 m/px |
| z17 | 0.97 m/px |
| z18 | 0.49 m/px |

DEM5A 是 5 m 网格，也就是说 z15 已经略微过采样了 `slopemap` 与 `hillshademap` 所依据的高程数据。GSI 把 `slopemap` 停在 z15 是信息量上限，不是随意设定的封顶。

由此可知：展示中把坡度图放大到 z17 时的模糊，缓存更深层级解决不了，把同一张**已渲染**的栅格改到客户端绘制同样解决不了。只有换用更细的 DEM，或者对**高程**做插值后再计算坡度（而不是对渲染后的图像做插值），才会改变。

## 高程解码，已用项目数据交叉验证

`dem5a_png` 以 `x = R*65536 + G*256 + B` 编码高程：小于 2^23 时 `h = x*0.01`，大于时 `h = (x - 2^24)*0.01`，`(128,0,0)` 保留为无数据。

横滨一块 z15 瓦片解码得 min/median/max 为 3.6 / 36.8 / 72.4 m，65,536 个像元中有 186 个无数据。`building_risk.geojson` 在更小的建筑覆盖范围上记录的 `elevation_m` 为 7.0 / 28.2 / 64.4 m。两者量程一致，说明解码正确。

## 库体积实测

| | 原始 | gzip | brotli |
|---|---|---|---|
| CesiumJS 1.143 `Cesium.js` | 5.64 MB | 1.64 MB | 1.31 MB |
| MapLibre GL 5 | 1.0 MB | 269 KB | — |
| deck.gl 9（UMD，全量图层） | 1.6 MB | 459 KB | — |
| MapLibre CSS | 72 KB | 10 KB | — |

Cesium 的 `Build/Cesium` 目录在磁盘上是 22 MB，但 `Assets`（4.6 MB）、`Workers`（1.3 MB）、`ThirdParty`（1.1 MB）是按需加载而非首屏加载。

传输量合计（gzip）：**Cesium 1.64 MB，MapLibre + deck.gl 738 KB。** deck.gl 那个数字是 UMD 全量包，用打包器只引入实际使用的图层会更小。

## CesiumJS 相关发现

- **不需要依赖 Cesium Ion。** 设置 `baseLayerPicker: false` 与 `geocoder: false`，并提供 GSI 影像和自定义地形提供器，所有请求都留在 GSI。若用默认配置，它会去连 Cesium 的服务器。
- **GSI 高程的地形提供器约 50 行。** 社区模块面向 1.104 之前的构造器 API，被引用最多的那个已归档，因此自己写比依赖它们更合适。
- **场景模式 morph 无法用作 2D/3D 切换。** `morphTo2D` 与 `morphTo3D` 会为新投影重建全部几何；在 2,128 栋挤出建筑下卡顿不可接受。改为设置 `scene3DOnly: true` 并通过相机切换——俯视加 `switchToOrthographicFrustum()`——是即时的，视觉上等价。
- **大批量几何不该走 Entity API。** `GeoJsonDataSource` 配 `CLAMP_TO_GROUND` 处理 2,128 个多边形，加载慢、切换也慢。改用数据中已有的 `elevation_m`，以绝对高程构建单个批处理 `Primitive`，完全免除地形采样，加载无可感延迟。
- **`requestRenderMode: true` 会让静止时的帧率成为无意义指标**，因为静止帧被刻意跳过。对分析型视图而言这仍是正确设置。
- **Cesium 的瓶颈在主线程 CPU。** 瓦片选择、剔除、绘制命令生成每帧都在单线程 JavaScript 中执行。在 M4 Max 上实测帧率仍低于 60，增加核心数无济于事。

## MapLibre + deck.gl 相关发现

- **在同一台机器、同一地区、同一数据下，帧率明显优于 Cesium。**
- **矢量底图超出源层级上限后仍然锐利。** `experimental_bvmap` 只到 z16，再放大是几何缩放而非像素缩放，因此在栅格影像已经模糊的区域，线划和注记依然清晰。
- **具备 GPU 聚合能力，Cesium 没有对等物。** `HexagonLayer` 在 GPU 上按 `risk_score` 对建筑集合做聚合。
- **基于 GSI 高程的地形在 spike 中未能跑通。** 在 spike 自身代码里发现并修复了三个缺陷：并发瓦片请求间共享解码 canvas 造成的竞态；缺少 `colorSpaceConversion: "none"`，导致浏览器改动了作为数据而非颜色的 RGB 值；以及编码不匹配——处理器输出 terrarium，而数据源声明的是 MapLibre 默认的 `mapbox` 编码。三处全部修复后地形依然渲染错误，原因未查明。**这是 spike 的局限，不能作为 MapLibre 能力不足的证据。**
- **deck.gl 的挤出不会自动贴合地形。** 挤出多边形从高程 0 起算，开启地形后建筑会陷入或浮于坡面之上。Cesium 用绝对高程天然对齐。修正这一点是实实在在的额外工作量。

## 最初设想下的地下视角

当时把它当作决定性需求，在那个设想下 Cesium 完胜。设想本身是错的，下一节说明原因。此处保留记录，因为能力对比本身仍然准确。

Cesium 对地下视角原生支持：相机可下降到地形之下且导航行为随之调整，`globe.translucency` 让地表半透明以透出地下几何。它是钻孔、隧道、地质层可视化的标准工具。

MapLibre 的相机是地表相机，地形网格会遮挡其下方的一切。可以在负高程绘制地下几何，但无法从下方观看。

地下可视化确实是产品需求——Synthetic Data Layer 的入口证据 `geo_pfn` 稀疏地下预测本身就在地下。错的是假定这个需求意味着「把广域相机潜到地下」。

## 地下数据究竟在哪里看

评估后期分离出两个场景，正是这个分离确定了最终选型。

**地图上——离散的网状设施。** 管线、电缆、设施回答的是"它从哪走到哪"，需要区域上下文。压低相机（`setMaxPitch(85)`，MapLibre 默认为 60）并把建筑与底图调半透明，就能直观看到城市地下的管网。相机始终不进入地下：deck.gl 暴露了 `depthTest`、`depthMask`、`depthCompare`，管网图层可以穿透地表绘制。此场景关闭 3D 地形，从而移除那层会遮挡负高程几何的网格。

**局部场景——连续场。** 土质、地层、预测属性回答的是"这一点下面是什么"，需要剖切和局部精度。用户框选一个区块，切换进独立的 Three.js 场景，其中同时包含地上与地下要素。土壤预测与不确定性只在这里出现。

在广域地图上渲染地层和 3D 剖面无法阅读、操作不便，本就不该放在那里。

## 决定

**地图采用 MapLibre GL JS + deck.gl，场址视图采用独立的 Three.js 场景。不采用 CesiumJS。**

排除 Cesium 不是因为能力不足——它的地下支持在候选中确实最好——而是因为它所满足的需求，形态与最初设想的不同。在广域地图上把相机潜入地形之下视觉冲击很强，但它回答的是一个用户不会问的问题。一旦把地下工作拆成"地图层面的管网查看"和"局部场址场景"，地图栈就不再需要降到地表以下，场址场景也不再需要地理空间球体引擎。

选型依据是 MapLibre 的实测优势——体积、帧率、2D 操作性、底图锐度——外加一条战略理由：`flutter-maplibre-gl` 的存在意味着图层与样式模型可以在将来迁移到原生平板客户端。Cesium 在原生端没有对等物。

把场址视图分离到独立的 Three.js canvas，还同时消除了多个问题：三个库不共享 WebGL 上下文；使用局部米制坐标而非墨卡托世界坐标，float32 精度绰绰有余；垂直夸大变成一个简单的缩放系数；剖切可直接使用原生裁剪平面；体数据的 raymarching 被限制在局部包围盒内。

## 需带入实施阶段的后果

- **审计链在进入场址场景后不能断裂。** 局部场景的坐标是相对场址原点的米值，因此原点、旋转、高程基准与默认夸大系数必须由 API 作为一等数据返回，绝不能在客户端硬编码。场景中选中的钻孔，仍须能回溯到真实坐标、高程基准、来源与采样时间。
- **地下管网数据尚不存在。** 地图层面的场景在技术上已无障碍，但受数据阻塞。
- **MapLibre 上的地形仍未验证。** 地下场景用不到它，但地表上下文仍然需要，而 spike 未能解决。

## 已核实但非实测的相关结论

- **排除 SolidJS。** 其最新稳定版为 v1.9.0（2024-09-24）；v2.0.0-beta.0 始于 2026-03 且尚未转正。Svelte 与 React 均保持周级至月级的发布节奏。
- **否决用 JavaScript 替换 Python 后端。** Web 层为 380 行（`api.py` 加 `data_service.py`），而管线与地理空间层为 1,951 行，依赖 rasterio/GDAL、pyproj/PROJ、numpy 与 Pillow，这些在 JavaScript 生态中没有对等物。可迁移的仅占后端约 14%，且无论如何 Python 进程都必须保留。

- **Flutter 是暂缓而非排除。** 此前的排除以 Flutter Web 承载 CesiumJS 为前提，而 CanvasKit 独占画布使该前提不成立。到了现场使用的原生平板，这个前提本身就不适用。`flutter-maplibre-gl` 活跃但尚未到 1.0 且体量较小；ArcGIS Maps SDK for Flutter 的开发构建带水印，生产环境需要授权字符串与 Esri 账号，且 ArcGIS Location Services 按调用量计费，这与零付费数据服务的定位冲突。现场客户端与展示是两个不同的产品，用 Capacitor 包装同一个 Web 应用是更廉价的第一步。
