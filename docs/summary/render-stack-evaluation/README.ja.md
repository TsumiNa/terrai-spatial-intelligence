# レンダリングスタック評価：CesiumJS と MapLibre + deck.gl

[English](README.md) | [日本語](README.ja.md) | [中文](README.zh.md)

- 日付：2026-07-20
- 状態：決定済み——地図は MapLibre GL JS + deck.gl、場址シーンは独立した Three.js
- 範囲：予定されているフロントエンド再構築に先立つ、ブラウザ側レンダリングスタックの評価

## この評価を行う理由

Leaflet は 2D ラスタタイルしか描画しません。ロードマップには 3D 地形と地下可視化が入っており、Leaflet では表現そのものが不可能です。同一地域・同一の `building_risk.geojson`（建物 2,128 件）・同一の GSI ソースに対し、2 つの候補スタックを使い捨ての spike として構築しました。

本書は測定結果と、その測定が導いた決定の記録です。

## GSI ソースの上限

2 つのデモ矩形に対して直接プローブした結果です。

| レイヤ | 提供される最深ズーム | 備考 |
|---|---|---|
| `slopemap` | z15 | z16 は 404 |
| `hillshademap` | z16 | z17 は 404 |
| `seamlessphoto` | z18 | z19 は 404 |
| `ort` | z18 | 同一ズームで seamlessphoto の約 17 KB に対し約 57 KB/タイル |
| `dem5a_png` | z15 | 数値標高。両地域とも利用可能 |
| `dem_png` | z14 | DEM10B のフォールバック |
| `experimental_bvmap` | z16 | ベクタタイル。774 レイヤの公式スタイルが公開されている |

GSI タイルはすべて `access-control-allow-origin: *` を返すため、ブラウザ側での画素デコードが可能です。

## 地上分解能と、ラスタ傾斜量図を鮮明化できない理由

北緯 35.45° において：

| ズーム | 分解能 |
|---|---|
| z15 | 3.89 m/px |
| z16 | 1.95 m/px |
| z17 | 0.97 m/px |
| z18 | 0.49 m/px |

DEM5A は 5 m グリッドであり、z15 の時点で `slopemap` と `hillshademap` の元となる標高データを既にわずかにオーバーサンプリングしています。GSI が `slopemap` を z15 で止めているのは情報量の限界であって、恣意的な上限ではありません。

帰結として、展示で傾斜を z17 に拡大したときの不鮮明さは、より深いズームをキャッシュしても解消せず、同じ描画済みラスタをクライアント側で描いても解消しません。解消するのはより細かい DEM を使う場合か、描画済み画像ではなく標高を補間してから傾斜を計算する場合だけです。

## 標高デコードのプロジェクトデータによる検証

`dem5a_png` は `x = R*65536 + G*256 + B` で標高を符号化し、2^23 未満は `h = x*0.01`、それ以上は `h = (x - 2^24)*0.01`、`(128,0,0)` を無データとして予約しています。

横浜の z15 タイル 1 枚をデコードすると min/median/max は 3.6 / 36.8 / 72.4 m、無データ画素は 65,536 中 186 でした。`building_risk.geojson` はより狭い建物範囲について `elevation_m` を 7.0 / 28.2 / 64.4 m で保持しています。範囲は整合しており、デコードは正しいと判断できます。

## ライブラリ容量の実測

| | 生 | gzip | brotli |
|---|---|---|---|
| CesiumJS 1.143 `Cesium.js` | 5.64 MB | 1.64 MB | 1.31 MB |
| MapLibre GL 5 | 1.0 MB | 269 KB | — |
| deck.gl 9（UMD、全レイヤ） | 1.6 MB | 459 KB | — |
| MapLibre CSS | 72 KB | 10 KB | — |

Cesium の `Build/Cesium` はディスク上 22 MB ですが、`Assets`（4.6 MB）、`Workers`（1.3 MB）、`ThirdParty`（1.1 MB）は初回描画時ではなく必要時に読み込まれます。

転送量の合計（gzip）：**Cesium 1.64 MB、MapLibre + deck.gl 738 KB。** deck.gl の値は UMD 全量パッケージであり、バンドラで使用レイヤのみを取り込めばさらに小さくなります。

## CesiumJS に関する所見

- **Cesium Ion への依存は不要です。** `baseLayerPicker: false` と `geocoder: false` を設定し、GSI の画像とカスタム地形プロバイダを与えれば、すべてのリクエストが GSI に留まります。既定構成のままでは Cesium のサーバへ接続します。
- **GSI 標高用の地形プロバイダは約 50 行です。** コミュニティのモジュールは 1.104 以前のコンストラクタ API 向けで、最も参照されるものはアーカイブ済みのため、依存するより自作が適切です。
- **シーンモードの morph は 2D/3D 切替として使えません。** `morphTo2D` と `morphTo3D` は新しい投影のために全ジオメトリを再構築します。挤出した建物 2,128 件では引っ掛かりが許容できませんでした。`scene3DOnly: true` を設定しカメラ側で切り替える方式——真上からの視点と `switchToOrthographicFrustum()`——は即時で、見た目も等価です。
- **大量ジオメトリに Entity API は不適切です。** `GeoJsonDataSource` に `CLAMP_TO_GROUND` を付けてポリゴン 2,128 件を扱うと、読み込みも切替も遅くなりました。データに既にある `elevation_m` を用いて絶対高で 1 つのバッチ `Primitive` を構築すると、地形サンプリングが完全に不要になり、体感できる遅延なく読み込まれます。
- **`requestRenderMode: true` は静止時の FPS を無意味な指標にします。** 静止フレームを意図的に描画しないためです。分析用ビューとしてはこれが正しい設定です。
- **Cesium は主スレッドで CPU 律速です。** タイル選択、カリング、描画コマンド生成が毎フレーム単一スレッドの JavaScript で走ります。M4 Max でも FPS は 60 未満に留まり、コア数の増加は効きません。

## MapLibre + deck.gl に関する所見

- **同一マシン・同一地域・同一データで、フレームレートは Cesium より明確に優れます。**
- **ベクタ基図はソースの上限を超えても鮮明さを保ちます。** `experimental_bvmap` は z16 までですが、それ以降は画素ではなくジオメトリが拡大されるため、ラスタ画像がぼやける領域でも線と注記が保たれます。
- **GPU 集約が利用でき、Cesium に対等物はありません。** `HexagonLayer` が建物集合を `risk_score` で GPU 上集約します。
- **GSI 標高による地形は spike 上で未解決です。** spike 自身のコードで 3 つの欠陥を発見・修正しました。並行タイル要求間で共有デコード canvas が競合していたこと、`colorSpaceConversion: "none"` の欠落により色ではなくデータである RGB 値がブラウザに改変されていたこと、そしてハンドラが terrarium を出力する一方でソースが MapLibre 既定の `mapbox` 符号化を宣言していた不整合です。3 件すべてを修正しても地形は正しく描画されませんでした。原因は特定できていません。**これは spike の限界であり、MapLibre の限界が示されたわけではありません。**
- **deck.gl の挤出は地形に自動追従しません。** 挤出ポリゴンは標高 0 から始まるため、地形を有効にすると建物が斜面に沈むか浮きます。Cesium は絶対高でそのまま整合しました。この是正には実際の追加工数が必要です。

## 当初の枠組みでの地下視点

これが決定的要件だと想定しており、その枠組みでは Cesium が明確に優位です。枠組み自体が誤りであったことは次節で述べます。能力比較としては依然正確なため、記録として残します。

Cesium は地下視点に標準で対応します。カメラが地形の下へ降り、それに応じて操作挙動も調整され、`globe.translucency` が地表を減光して地下ジオメトリを見せます。ボーリング、トンネル、地質層の可視化における標準的な手段です。

MapLibre のカメラは地表カメラであり、地形メッシュがその下にあるものを遮蔽します。負の標高に地下ジオメトリを描くことはできますが、下から見ることはできません。

地下可視化は製品要件です——Synthetic Data Layer の入口証拠である `geo_pfn` の疎な地下予測が地下そのものだからです。誤っていたのは、その要件が「広域カメラを地下へ潜らせること」を意味すると想定した点です。

## 地下データを実際に見る場所

評価の終盤で 2 つの場面を分離しました。この分離が決定を確定させました。

**地図上——離散的なネットワーク設備。** 管路、ケーブル、施設は「どこからどこへ通っているか」という問いであり、広域の文脈を必要とします。カメラを下げ（`setMaxPitch(85)`、MapLibre の既定は 60）、建物と基図を半透明にすれば、市街地の地下ネットワークを直感的に把握できます。カメラが地下へ入ることはありません。deck.gl は `depthTest`、`depthMask`、`depthCompare` を公開しており、ネットワーク層を地表越しに描画できるためです。この場面では 3D 地形を無効にし、負の標高のジオメトリを遮蔽するメッシュを取り除きます。

**局所シーン——連続場。** 土質、地層、予測物性は「この地点の下に何があるか」という問いであり、断面切断と局所的な精度を必要とします。利用者が区画を矩形選択し、地上と地下の両方を含む独立した Three.js シーンへ切り替えます。土質予測と不確実性の表示はここにのみ現れます。

広域地図上に地層や 3D 断面を描画しても読み取れず、操作もしにくく、そこに置くべきものではありません。

## 決定

**地図は MapLibre GL JS + deck.gl、場址ビューは独立した Three.js シーン。CesiumJS は採用しない。**

Cesium を除外したのは能力不足のためではありません。地下対応は候補中で実際に最良でした。除外の理由は、それが満たす要件の形が当初の想定と異なっていたことです。広域地図でカメラを地形の下へ潜らせるのは視覚的に印象的ですが、利用者が問わない問いに答えています。地下作業を「地図上のネットワーク閲覧」と「局所の場址シーン」に分けた時点で、地図スタックに地表以下へ降りる必要はなくなり、場址シーンに地理空間グローブエンジンも不要になります。

スタック選定は MapLibre の実測上の優位——容量、フレームレート、2D の操作性、基図の鮮明さ——に加え、戦略的理由が 1 つあります。`flutter-maplibre-gl` が存在するため、レイヤとスタイルのモデルを将来ネイティブのタブレットクライアントへ移せます。Cesium にネイティブの対等物はありません。

場址ビューを別の Three.js キャンバスに分けることで、複数の問題が同時に解消します。3 つのライブラリが WebGL コンテキストを共有しない、メルカトル世界座標ではなく局所メートル座標を使うため float32 の精度で十分、垂直誇張が単純なスケール係数になる、断面切断にクリッピング平面をそのまま使える、ボリュームのレイマーチングを局所の境界箱に限定できる、という点です。

## 実装へ持ち越す帰結

- **監査の連鎖は場址シーンに入っても途切れてはなりません。** 局所シーンの座標は場址原点からのメートル値であるため、原点、回転、標高基準、既定の誇張率は API が一級のデータとして返すべきで、クライアントに直書きしてはいけません。シーン内で選択したボーリングは、実座標、標高基準、出典、採取日時まで辿れる必要があります。
- **地下ネットワークのデータはまだ存在しません。** 地図上の場面は技術的には可能ですが、データ待ちです。
- **MapLibre 上の地形は未検証のままです。** 地下の場面には不要ですが、地表の文脈表現には依然として必要であり、spike では解決できませんでした。

## 確認済みだが実測ではない関連事項

- **SolidJS は除外しました。** 最新の安定版は v1.9.0（2024-09-24）で、v2.0.0-beta.0 は 2026-03 以降も安定版に至っていません。Svelte と React はいずれも週次から月次でリリースしています。
- **Python バックエンドを JavaScript へ置き換える案は却下しました。** Web 層は 380 行（`api.py` と `data_service.py`）である一方、パイプラインと地理空間層は 1,951 行あり、rasterio/GDAL、pyproj/PROJ、numpy、Pillow に依存します。これらに JavaScript エコシステムの対等物は存在しません。移行できるのは後端の約 14% に過ぎず、いずれにせよ Python プロセスは残ります。
- **Flutter は除外ではなく保留です。** 以前の除外は Flutter Web が CesiumJS を内包する前提でしたが、CanvasKit がキャンバスを占有するためそれは成立しません。現場のネイティブタブレットではその前提自体が当てはまりません。`flutter-maplibre-gl` は活発ですが 1.0 未満で規模も小さく、ArcGIS Maps SDK for Flutter は開発ビルドに透かしが入り、本番にはライセンス文字列と Esri アカウントが必要で、ArcGIS Location Services は従量課金です。これは無償データサービスという位置づけと衝突します。現場クライアントは展示とは別の製品であり、同一 Web アプリを Capacitor で包むほうが安価な第一歩です。

## 再現について

spike は使い捨てであり、リポジトリ外に置かれています。上記の数値はすべて、ベンダーの宣伝ではなく、GSI エンドポイントへの直接プローブ、ダウンロードしたライブラリビルドの実測、または同梱ライブラリのソース確認から得たものです。
