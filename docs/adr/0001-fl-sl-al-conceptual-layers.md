# ADR-0001：采用 FL → SL → AL 作为 TerrAI 概念架构

- 状态：Accepted（Factor of Concept）
- 日期：2026-07-20
- 决策者：TerrAI
- 范围：产品叙事、Demo 信息架构与后续开发边界

## Context

TerrAI 当前 Prototype 已合并多个开源空间数据和三个应用方向，但现有“统一数据底座”说法没有清楚回答：哪些值是观测、哪些是确定性加工、哪些将来由稀疏预测生成、应用评分应该消费什么。

战略文档 v4 要求建设可跨客户复用的 engine core、标准 delivery layer 和有限的 application layer；Cyber Port/羽田 Demo 只证明“稀疏观测下预测地层参数”这一入口能力，不能被当成完整应用。用户进一步将该战略具体化为 Foundation Data Layer、Synthetic Data Layer 与 Application Layer。

`geo_pfn` 又表明模型性能随上下文密度、特征组和模型规模改变；这不支持“一次训练、处处填满”的单模型结构，而支持一个带适用性、校准和拒答能力的场景模型群。

## Decision

采用三层概念架构：

1. **FL** 保存公开、商业和客户授权的现实观测，以及不改变观测语义的确定性加工；允许多尺度 missing。
2. **SL** 在 FL 上非破坏性地生成经过验证的稀疏补值/增强；observed、synthetic、unresolved 必须保持概念可区分，并携带不确定性和适用边界。
3. **AL** 将 FL/SL 证据转为场景规则、排序和行动出口；不能把 synthetic 隐藏为事实。

本决策同时规定：

- 不是所有 missing 都应该补值，法律、权属、正式工程结论等应保持 unknown 并触发权威流程。
- 当前空间 Demo 没有已经验证的地表 SL。现有风险分、适宜分、容量代理和联合分属于 AL 的启发式计算。
- Google Satellite Embedding 属于 FL 的外部观测表征，不属于 TerrAI 生成的 SL。
- `geo_pfn` 只作为稀疏迁移、模型选择和不确定性的机制证据，不外推为坡地、道路或光伏精度。

## Alternatives considered

### A. 继续使用“统一数据底座 + 多应用”

优点是简单；缺点是会继续混淆事实、代理和模型补值，也无法说明稀疏预测如何成为跨应用资产。拒绝。

### B. 每个应用各自维护补值模型

优点是短期交付快；缺点是客户学习和验证留在单项目中，违背战略文档“下一次部署应更少定制”的 scaling test。拒绝作为总体架构。

### C. FL → SL → AL

能把客户/开放数据积累、稀疏预测能力与应用收入出口分开演进，并把不确定性和审计边界放在应用之前。采用。

## Consequences

正面影响：

- TerrAI 的可复用资产从“共享地图代码”升级为 FL 数据积累和 SL 模型能力。
- 新应用可以复用 SL，不必重新把缺测问题埋在业务评分中。
- observed/synthetic 分离，使审计、不确定性和人工复核成为产品结构而非免责声明。
- 模型群和 abstention 有了明确位置，能容纳不同客户与稀疏度。

成本与风险：

- 产品必须接受部分区域继续 unknown，不能以“全覆盖地图”作为唯一完成标准。
- SL 的跨场景复用需要严格验证；地下证据不能直接迁移到地表。
- 后续开发需要处理版本、权限、模型选择和 lineage，但本 ADR 不预先决定实现方式。

## Non-goals of this ADR

本 ADR 不定义数据结构、API、数据库、模型注册、调度、部署或客户权限方案。这些属于后续 Factor of Develop ADR。

## Validation

本概念重构通过以下方式验收：

- 工程文档中存在唯一、明确的 FL/SL/AL 定义与非目标；
- Demo 提供三层架构视图并如实显示当前 FL/SL/AL 成熟度；
- 现有应用继续运行，且不把当前启发式结果误标为 SL；
- 三语界面保持一致；
- PR 记录原因、步骤、证据变化和后续开发问题。
